## Submission rules (both homeworks)

1. **Your work must be in a public GitHub repo of your own.** Don't fork the reference and call it done — create a fresh repo so the commit history shows *your* thinking.
2. **Post on Skool.** Both submissions get a Skool post in the class group. Format:
   - One-line summary of what you built/learned
   - Repo link
   - Demo video / screenshots (HW2 only — required)
   - Three things that surprised you or that you got stuck on
3. **Do not blindly vibe-code.** We want to see thinking, not just generated code that runs once.
   - For each non-trivial decision, drop a short note in your README explaining *why* you chose that approach. One or two sentences is enough — but they have to be yours.
   - If you used Claude Code / Cursor / Codex / etc., that's fine and expected. The bar is: you understood every line you committed and could defend it if asked.
4. **Keep secrets out of git.** `.env` files, API keys, DB URLs — never commit. Use `.env.example` for the shape, `.gitignore` for the real one.

---

## What "good" looks like

Both homeworks are graded on the same three things:

1. **Does it run?** Fresh clone → follow your README → working agent / completed workbook.
2. **Did you think?** Your README, commit messages, and code comments show that you understood the problem, not just pattern-matched on the demo.
3. **Did you teach it back?** Your Skool post should help the next person in the cohort. Concrete gotchas, not "great class, learned a lot."
